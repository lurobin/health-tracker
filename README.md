# Health-Tracker · CLI-Native AI 健康管理系统

不是 app，不是 web，不是 wearable。用 **Markdown + AI Agent + 规则引擎** 在终端里运行一整套个人健康管理系统。覆盖饮食管理、运动训练、膝盖康复、扁平足功能恢复、睡眠优化等多个领域。

> 🏆 **2026 小米 MiMo 百亿创造者激励计划 参赛项目**

---

## 亮点

### 1. CLI 原生 —— 极简基础设施

整个系统运行在命令行中。AI Agent 读取本地 Markdown 文件 → 分析 → 写回 Markdown 文件。没有数据库、没有服务器、没有 SaaS。

```
用户终端
  │
  ├── AI Agent (Claude Code / Gemini CLI)
  │     ├── 读取模板 → 生成每日文件
  │     ├── 读取数据 → 分析推理
  │     └── 写入反馈 → 锁定闭环
  │
  └── 本地文件系统 (Markdown)
        ├── Guides/        模板层
        ├── Private/Diary/ 运行时数据
        └── Data/          知识层
```

### 2. 医学知识工程化

不是扔给用户一篇文献让 TA 自己读。我们把专业内容**编码为可执行的规则系统**：

| 领域 | 工程化产出 | 来源 |
|:---|:---|:---|
| 膝康复 | Jill Cook 腱病分期（Stage 0→3）+ 等长镇痛协议 | 《运动损伤临床指南》|
| 扁平足 | 阶段零→三完整路线（TA 松解 → 足弓感知 → 绞盘效应） | 《练脚掌是最好的复健！》|
| 动作筛选 | Traffic Light 膝负荷分级（20+ 动作标记 🟢🟡🔴） | NASM-CES |
| 步态纠正 | 髋主导模式 + Mayo Clinic 8 条图像清单 | Gait & Posture 2019 |
| 足弓评估 | 舟骨沉降测试 + 足弓指数 + 9 张标准化拍照规范 | Brody 1982, Murley 2009 |

### 3. 规则 + AI 混合决策

不是纯 prompt 也不是纯代码——两者结合：

```
压痛测试(阳性) ──→ [规则] 强制执行休息日
晨起RHR ≥76    ──→ [规则] 完全休息
心慌手抖       ──→ [AI 推理] 是否低血糖？→ 建议快碳
训练疲劳/不想练 ──→ [AI 判断] 执行最低合格线
```

**强度第一准则**：压痛测试优先级高于一切心率指标。系统级的"安全断路器"。

### 4. 自我迭代闭环

每日 `lock` 操作触发完整反馈循环：

```
数据解读 → 问题定位(≤2个) → 调整建议(≤3条) → 生成次日文件
```

月总结不仅看体重趋势和膝盖进展，还检查外部 AI review 的**逐条落地状态**。系统不是一次性输出——它在持续追踪自己的建议是否被执行、是否有效。

### 5. 诚实的数据记录

多因子推理链确保**异常状态优先于常规流程**：

```
低血糖(心慌手抖) → 快碳需求 → 神经空虚 → 体重波动 → 膝盖状态
```

用户标注"心慌手抖"时，系统优先判断低血糖而非照常反馈饮食。高优先级信号会短路后续判断——这不是模板化的格式化输出。

### 6. 系统与用户共同成长

系统接受多种格式的外部输入：知乎深度文章、专业 PDF、文献摘要、第三方 AI review。每一条经过评估后，有选择地吸收到 `Guides/` 模板和 `Data/knowledge-base.md` 中，优化下期训练内容。

18 本运动医学/营养学参考文献被逐步消化为可执行的规则和指标。系统的"知识层"在持续扩展。

### 7. 隐私即特性

个人数据全部存储在本地 `Private/` 目录。AI 读取本地文件，处理结果也写入本地。不上传、不共享、不 SaaS。

这套设计本身就是加分项——在 AI 平台评审眼中，这证明你从一开始就考虑了隐私和合规。

```
┌──────────────────────────────────────────────────────┐
│                    AI Agent 层                        │
│     Claude Code / Gemini CLI · 读取→分析→生成        │
│     长链推理 · 多因子决策 · 自适应调整               │
├──────────────────────────────────────────────────────┤
│                    模板层                            │
│  Daily-Template.md   训练模板   动作库   任务线       │

│                    运行时数据                         │
```

## 核心功能

### 每日运行闭环

| 触发词 | 行为 |
|:---|:---|
| `look` / 查看 | 读取当日数据 → 即时反馈 |
| `lock` / 总结 | 反馈 + Gap 分析 → 自动生成次日文件 → 锁定当日 |

### 自适应训练系统

**三天轮换**：上肢推(A) / 上肢拉(B) / 全身循环(C)，配合核心四选一每日轮换

**模块状态管理**：活跃(每天) → 降频(隔天) → 维持(周1-2次) → 休眠(暂停)，根据身体反馈自动流转。

**最小有效剂量**：连续 2 天 RHR 升高或主观疲劳 → 执行最低合格线（核心×1组 + 上肢×1组 + 扳机），不追完整训练。

### 多因子决策链

```
低血糖 → 快碳 → 神经空虚 → 体重波动 → 膝盖状态
```

每个优先级可短路后续判断。高强度优先级信号未处理前，不进入下一层。

### 验证闭环（🟢🟡🔴）

| 信号 | 标准 | 处理 |
|:---|:---|:---|
| 🟢 | 周降 0.3–0.6kg | 继续 |
| 🟡 | 周降 >0.8kg + 疲劳 | 补碳+10g |
| 🔴 | 日降 >1kg + RHR 升高 | 停练+补碳 |

### Traffic Light 疼痛响应

全部动作按膝负荷分为 🟢 安全 / 🟡 谨慎 / 🔴 替换三级，疼痛日自动过滤和强制替换。

## 设计理念

**用户手动记录 > 设备自动数据**——手动记录本身就是知觉建立的过程。

**结构化模板 > 自由 prompt**——系统不依赖特定 AI 模型，换一个 Agent 照常运行。

**知识工程化**——每条规则对应可追溯的文献来源，非经验主义。

**隐私即特性**——零服务器、零上传，AI 在本地完成全部读写。
